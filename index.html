Juego tipo Arkanoid

1. Introducción

1.1 Propósito

Este documento tiene como propósito definir los requisitos del sistema para el desarrollo de un videojuego del tipo Arkanoid. Está destinado a los desarrolladores, diseñadores, evaluadores y cualquier parte interesada en comprender las funcionalidades y restricciones del sistema.

1.2 Alcance

El producto es un juego arcade en el que el jugador controla una paleta para evitar que una pelota caiga, haciéndola rebotar contra bloques situados en la parte superior de la pantalla. El objetivo es eliminar todos los bloques para avanzar de nivel. El juego está orientado a un solo jugador y se ejecutará en plataformas de escritorio o navegadores web.

1.3 Definiciones

  • UI: Interfaz de Usuario
  • FPS: Fotogramas por segundo
  • HTML5: Lenguaje de marcado para desarrollo web

1.4 Referencias

  • IEEE Std 830-1998, IEEE Recommended Practice for Software Requirements Specifications
  • Documentación de HTML5, CSS3 y JavaScript (para versiones web)

1.5 Visión general del documento

Este documento sigue la estructura recomendada por el estándar IEEE 830-1998 para especificaciones de requisitos de software. Está dividido en secciones que cubren la descripción general del producto, los requisitos funcionales, no funcionales y de interfaz, así como supuestos, restricciones y posibles requisitos futuros. Esta organización permite una comprensión clara y estructurada del sistema a desarrollar.

2. Descripción General

El producto a desarrollar es un videojuego tipo Arkanoid, un juego clásico de estilo arcade en el que el jugador controla una paleta ubicada en la parte inferior de la pantalla, con el objetivo de mantener en juego una pelota que rebota y destruye bloques ubicados en la parte superior. A medida que el jugador avanza, los niveles presentan mayor dificultad con patrones de bloques más complejos, aumento en la velocidad de la pelota y posibles elementos especiales. El juego se plantea como una aplicación de entretenimiento para un solo jugador, con una estética retro y mecánicas simples pero adictivas, enfocada tanto en usuarios nostálgicos de los juegos clásicos como en nuevos jugadores que buscan una experiencia casual.

2.1 Perspectiva del producto

El juego Arkanoid a desarrollar es una aplicación independiente, diseñada como un sistema autocontenible que se ejecutará localmente o en la web. Está compuesto por los siguientes módulos:
  • Motor de juego
  • Interfaz de usuario
  • Gestor de niveles
  • Módulo de entrada
  • Módulo de audio (opcional)
La arquitectura modular facilitará futuras extensiones como soporte multijugador, nuevos niveles y modos de juego.

2.2 Funciones del producto

El juego permite al usuario:
  • Controlar una paleta para hacer rebotar una pelota.
  • Destruir bloques al impactar la pelota contra ellos.
  • Ganar puntos por cada bloque destruido.
  • Avanzar de nivel tras destruir todos los bloques.
  • Perder vidas si la pelota no es interceptada.
  • Visualizar información en pantalla: puntuación, nivel, vidas.
  • Pausar, reiniciar o salir del juego desde el menú principal.

2.3 Características del usuario

Los usuarios esperados son jugadores casuales sin necesidad de experiencia previa. Se espera que tengan conocimientos básicos de uso del teclado o mouse y que busquen una experiencia de juego accesible, rápida y entretenida.

2.4 Restricciones

  • Compatibilidad con HTML5, CSS3 y JS
  • Diseño adaptado a resoluciones comunes
  • Compatibilidad con navegadores web que soporten HTML5, CSS y Javascript
  • Tiempo de respuesta inmediato (latencia baja)
  • El juego debe estar diseñado para resoluciones de pantalla comunes
  • Uso de tecnologías definidas (ej. HTML5 para versión web)

2.5 Suposiciones y dependencias

  • El usuario dispone de teclado y/o mouse funcional.
  • El entorno de ejecución es compatible con los requisitos mínimos.
  • No se requiere conexión a internet.

2.6 Requisitos futuros

Estas funcionalidades no se implementarán en la versión inicial, pero podrán incrementar la jugabilidad, accesibilidad y el valor del producto.
  • Modo multijugador en línea: Incorporar un modo de juego competitivo o cooperativo que permita a varios jugadores participar simultáneamente a través de Internet.
  • Editor de niveles personalizado: Permitir a los jugadores crear, guardar y compartir sus propios niveles personalizados mediante una interfaz gráfica amigable.
  • Integración con plataformas móviles: Adaptar el videojuego para que funcione en dispositivos móviles (Android e iOS), considerando aspectos como control táctil y optimización de rendimiento.
  • Integración con puntuaciones en la nube: Implementar una sección donde los jugadores puedan revisar su desempeño a través de gráficos, tiempos, puntajes y niveles completados.
  • Sistema de logros y recompensas: Incluir un sistema de trofeos o logros que motive al jugador a cumplir metas específicas dentro del juego.
  • Mecánicas de juego ampliadas: Incluir nuevos elementos como bloques con efectos especiales, potenciadores adicionales o enemigos dinámicos que se muevan por el escenario.

3. Requisitos Específicos

3.1 Requisitos Funcionales

  1. RF-01. Inicio del juego: El sistema deberá permitir al jugador iniciar una nueva partida mediante un botón de "Jugar" en la pantalla principal.
  2. RF-02. Movimiento de la plataforma: El sistema deberá permitir al jugador controlar la plataforma horizontalmente utilizando las teclas de dirección (flechas) o el mouse.
  3. RF-03. Rebote de la pelota: La pelota deberá rebotar automáticamente al impactar con la plataforma, los bloques o los bordes del escenario, modificando su trayectoria de forma realista.
  4. RF-04. Rompimiento de bloques: Cuando la pelota impacte un bloque, el sistema deberá eliminarlo visualmente y aumentar la puntuación del jugador.
  5. RF-05. Pérdida de vida: Si la pelota cae por debajo de la plataforma, el sistema deberá restar una vida al jugador y reiniciar la pelota en la plataforma.
  6. RF-06. Condiciones de victoria: El sistema deberá detectar cuando todos los bloques han sido eliminados y mostrar un mensaje de victoria.
  7. RF-07. Condiciones de derrota: El sistema deberá finalizar el juego si el jugador pierde todas sus vidas, mostrando un mensaje de derrota.
  8. RF-08. Sistema de puntuación: El sistema deberá mostrar y actualizar la puntuación del jugador en tiempo real conforme se destruyen bloques.
  9. RF-09. Potenciadores: El sistema deberá incluir bloques especiales que, al ser destruidos, generen potenciadores como: ampliar la plataforma, multiplicar la pelota o activar disparos.
  10. RF-10. Menú de pausa: El sistema deberá permitir pausar el juego mediante la tecla “Esc”, mostrando un menú con opciones para continuar, reiniciar o salir.

Requisitos No Funcionales

  • RNF-01. Rendimiento: El videojuego deberá ejecutarse con una tasa mínima de 30 FPS (cuadros por segundo) en equipos con especificaciones básicas.
  • RNF-02. Portabilidad: El videojuego deberá ser compatible con sistemas operativos Windows, Linux y MacOS.
  • RNF-03. Usabilidad: La interfaz del videojuego deberá ser intuitiva, permitiendo que un nuevo jugador comprenda la dinámica básica en menos de 1 minuto.
  • RNF-04. Tiempo de respuesta: Las acciones del jugador (como mover la plataforma o pausar el juego) deberán tener un tiempo de respuesta menor a 100 ms.
  • RNF-05. Seguridad: Los archivos del videojuego no deberán realizar escrituras en el sistema del usuario más allá de su carpeta de instalación y almacenamiento de partidas.
  • RNF-06. Mantenibilidad: El código fuente del videojuego deberá estar documentado y estructurado modularmente, facilitando futuras modificaciones o adiciones.
  • RNF-07. Accesibilidad: El juego deberá ofrecer una opción de alto contraste para personas con dificultades visuales.

3.2 Interfaces Externas

Interfaz de Usuario

La interfaz gráfica del videojuego deberá ser simple, intuitiva y estar orientada a una experiencia de juego fluida. A continuación, se describen los componentes principales de la interfaz:

  • Pantalla de inicio:
    • Botón "Jugar" para iniciar una partida.
    • Botón "Opciones" para acceder a configuración de sonido, controles y accesibilidad.
    • Botón "Salir" para cerrar el juego.
  • Pantalla de juego:
    • Visualización de la plataforma, pelota, bloques y potenciadores en tiempo real.
    • Indicador de puntaje actual.
    • Contador de vidas restantes.
    • Botón o tecla para pausar el juego (ej. tecla "Esc").
  • Pantalla de pausa:
    • Opción para continuar el juego.
    • Opción para reiniciar la partida.
    • Opción para volver al menú principal.
  • Pantalla de fin de juego:
    • Mensaje de “Victoria” o “Derrota” según el caso.
    • Botón para volver al menú principal o iniciar una nueva partida.
  • Controles del usuario:
    • Movimiento de la plataforma: teclas de flechas (←, →) o movimiento del mouse.
    • Pausa: tecla "Esc".
    • Confirmación y navegación en menús: teclado y mouse.

Interfaces de Hardware

El videojuego deberá ser compatible con:

  • Teclado: para controlar el movimiento de la plataforma y acceder a menús.
  • Mouse (opcional): para controlar la plataforma y navegar por la interfaz de usuario.
  • Gamepad (en futuras versiones): como medio alternativo para controlar el juego.

Interfaces de Software

El videojuego se ejecutará como una aplicación web en cualquier navegador.

  • No se requerirá conexión con bases de datos ni servicios en la nube en la versión inicial.
  • En versiones futuras, podrá integrarse con servicios de guardado en la nube o rankings en línea.

Interfaces de Comunicación

En la versión inicial del videojuego, no se establecerán comunicaciones con sistemas externos mediante red.

  • Ranking en línea: para comparar puntuaciones con otros jugadores (en versiones futuras).
  • Multijugador en línea: para competencia entre jugadores remotos mediante protocolo TCP/IP o WebSocket (en versiones futuras).
  • Actualización automática: el juego podrá conectarse a un servidor para verificar y descargar actualizaciones (en versiones futuras).

3.2 Funciones

Organización elegida:
Por objetos, ya que el videojuego se compone de entidades claramente identificables que interactúan entre sí. Cada objeto tendrá funciones específicas asociadas, lo que facilita el entendimiento del comportamiento del sistema.

Objeto: Jugador

  • Inicie una nueva partida desde el menú principal.
  • Controle el movimiento de la paleta mediante teclado o mouse.
  • Pause, reinicie o termine la partida.
  • Visualice su puntaje y número de vidas en pantalla.
  • Reciba una notificación al ganar o perder la partida.
  • Seleccione un nivel (en caso de tener niveles desbloqueados).

Objeto: Paleta

  • Mover la paleta horizontalmente en respuesta a las entradas del jugador.
  • Restringir el movimiento de la paleta dentro de los límites del escenario.
  • Detectar colisiones entre la paleta y la pelota.
  • Cambiar la dirección de la pelota tras la colisión según el punto de impacto.
  • Cambiar de tamaño si se aplica un potenciador.

Objeto: Pelota

  • Iniciar el movimiento de la pelota una vez que el jugador empiece el juego.
  • Detectar colisiones con la paleta, bloques, bordes del escenario y elementos especiales.
  • Rebotar en función del ángulo de impacto.
  • Perder una vida si la pelota cruza la parte inferior de la pantalla.
  • Duplicarse, acelerarse o ralentizarse si recibe ciertos potenciadores.

Objeto: Bloque

  • Detectar colisiones entre la pelota y los bloques.
  • Destruir el bloque y aumentar el puntaje del jugador al ser impactado.
  • Generar un potenciador aleatorio en ciertos casos.
  • Llevar un contador del número de bloques destruidos.

Objeto: Potenciador

  • Generar potenciadores aleatorios tras la destrucción de ciertos bloques.
  • Hacer caer el potenciador hacia la paleta.
  • Detectar si el jugador recoge el potenciador.
  • Aplicar el efecto correspondiente al juego (ej. paleta más grande, pelota de fuego, vida extra).
  • Eliminar el potenciador si no es recogido.

Objeto: Nivel

  • Cargar los datos del nivel actual (posición de bloques, paleta, dificultad).
  • Detectar la finalización del nivel al destruir todos los bloques.
  • Cargar el siguiente nivel automáticamente o mostrar una pantalla de transición.
  • Permitir el diseño e inclusión de nuevos niveles de forma modular.

Objeto: Interfaz de Usuario

  • Mostrar el menú principal con las opciones disponibles.
  • Mostrar información en pantalla durante el juego (puntaje, vidas, nivel).
  • Presentar mensajes de pausa, victoria o derrota según corresponda.
  • Registrar y procesar las acciones del jugador dentro del menú.
  • Permitir la navegación entre pantallas del sistema (inicio, opciones, juego, fin).

Objeto: Sistema de Puntuación

  • Aumentar el puntaje del jugador según el tipo de bloque destruido.
  • Mostrar el puntaje actual en pantalla en tiempo real.
  • Registrar el puntaje final al finalizar el juego.
  • (Futuro) Almacenar y ordenar puntajes en una tabla de rankings locales o en línea.

3.3 Requisitos de interfaz

  • Teclado: Movimiento izquierda/derecha, pausa.
  • Mouse: Movimiento de la paleta (opcional).
  • Pantalla: Menú principal, interfaz de juego, pantalla de fin de juego.

3.4 Requisitos de rendimiento

  • El sistema deberá tener soporte para pantallas desde una resolución de 720p.
  • El sistema deberá tener una latencia de entrada menor a 100 ms.

3.5 Restricciones de Diseño

  • Modularidad: El sistema debe estar dividido en módulos independientes (motor de juego, UI, entrada, audio, gestor de niveles) para facilitar el mantenimiento y la escalabilidad del producto.
  • Tecnologías estándar: La versión web debe desarrollarse utilizando tecnologías estándar como HTML5, CSS3 y JavaScript, sin dependencias en plugins externos.
  • Independencia de plataforma: La lógica del juego no debe estar acoplada a características específicas del sistema operativo o navegador.
  • Resoluciones fijas: El diseño gráfico debe ajustarse adecuadamente a resoluciones desde 1280×720 píxeles (HD) sin distorsión ni pérdida de funcionalidad.
  • Consistencia visual: El estilo visual debe mantenerse uniforme en todo el juego, con una estética retro coherente.

3.6 Atributos del Sistema

  • Disponibilidad: El juego estará disponible para ser ejecutado localmente en todo momento, sin requerir conexión a internet.
  • Usabilidad: La interfaz debe ser lo suficientemente intuitiva para que cualquier usuario pueda comprender el funcionamiento del juego sin necesidad de instrucciones complejas.
  • Mantenibilidad: La arquitectura modular debe permitir realizar correcciones, mejoras o extensiones sin afectar el resto del sistema.
  • Portabilidad: El sistema debe poder trasladarse fácilmente entre navegadores y sistemas operativos modernos sin modificaciones significativas.
  • Rendimiento: El sistema debe garantizar un rendimiento fluido, con una tasa de cuadros estable (≥ 30 FPS), baja latencia y tiempos de carga breves.
  • Seguridad: Aunque no maneja datos sensibles ni conexiones en línea, el sistema debe evitar fallos inesperados y manejar correctamente errores de entrada o estados inválidos.

4. Apéndice

Este documento podrá evolucionar conforme avance el desarrollo. Se podrá actualizar con nuevos requisitos derivados de pruebas, retroalimentación de usuarios o cambios en los objetivos del proyecto.
